# P0_EXECUTION_CHECKLIST.md

## 1. 目的
本清单只覆盖当前最关键、最影响后续推进判断的 P0 项。
目标不是“一次整理完全部”，而是先建立一个可信、统一、不自相矛盾的工程化主轴。

执行原则：
- 先基线，后扩展
- 先对齐，后美化
- 先消除冲突，后补充细节
- 先跑通最小认知闭环，后讨论更大蓝图

---

## 2. P0 总目标
完成以下四件事：

1. 建立正式六类基线文档
2. 把当前真实技术栈同步到基线
3. 处理占据关键位置的空壳文件
4. 明确哪些文件是主事实源，哪些只是交接材料或历史档案

---

## 3. P0-01：正式落位六类工程化基线文档

### 目标
建立项目的正式主轴，让后续所有推进都有统一判断依据。

### 涉及文件
- `PROJECT_CHARTER.md`
- `BOUNDARY_AND_SCOPE.md`
- `ARCHITECTURE_BASELINE.md`
- `DELIVERY_STAGES.md`
- `CHANGE_CONTROL.md`
- `QUALITY_GATES.md`

### 完成标准
- 六个文件都已存在且有内容
- 彼此之间没有明显冲突
- 术语统一
- 明确写入当前真实技术栈
- 能回答：
  - 为什么做
  - 做到哪
  - 怎么组织
  - 现在在哪个阶段
  - 偏了怎么改
  - 什么算达标

### 风险
- 继续沿用分散旧文档，导致“双重真相”
- 会话交接文档继续替代正式基线

### 动作
- 直接以六类基线作为正式主线
- 将原有内容按角色归并进去
- 不再让 handoff 文档承担正式基线职责

## 4. P0-02：同步当前真实技术栈

### 必须同步的变化
- `LiteLLM -> Higress`

### 必须检查并更新的位置
- `README.md`
- `TOOLS.md`
- `ARCHITECTURE_BASELINE.md`
- `PROJECT_CHARTER.md`
- `BOUNDARY_AND_SCOPE.md`
- `GOVERNANCE_GLOSSARY.md`
- `RISK_MODEL.md`
- `APPROVAL_POLICY.md`

### 完成标准
- 仓库核心文档中不再把 LiteLLM 写成当前模型路由主组件
- 新组件都被纳入系统角色图
- 审批、后台、自动化、AI 应用中枢职责有明确落位
- 不再出现“旧架构图主导，新架构口头补丁修正”的情况

### 风险

- 集成设计和治理边界继续漂移

- 明确：
  - Higress = 模型入口网关

## 5. P0-03：处理关键空壳文件

### 目标
清除最具有误导性的“假完成感”。

### 重点文件
- `BOOTSTRAP.md`
- `governance/RULE_HARDENING_POLICY.md`

### 判断规则
空文件如果占据关键命名位，会带来三种毒性：
1. 让人误以为此能力已定义
2. 让后续文档引用一个不存在的真相
3. 让阶段判断失真

### 完成标准
对每个关键空壳文件二选一：
- **补成最小有效版本**
- **明确删除 / 降级 / 延后**

### 建议处理
#### `BOOTSTRAP.md`
建议至少补成最小说明：
- bootstrap 的目的
- 触发时机
- 与 `BOOT.md`、`PROJECT_BOOTSTRAP_SPEC.md` 的关系
- 当前是否已实现
- 若未实现，处于什么阶段

#### `RULE_HARDENING_POLICY.md`
建议至少补成最小说明：
- 哪些规则值得优先硬化
- 何时从 prose 升级到 policy/schema/validator
- 谁负责同步
- 出现冲突时哪个版本优先

### 风险
- 继续把空壳当已完成模块引用
- 阶段推进误以为“规则硬化设计已经存在”

## 6. P0-04：明确主事实源与降级材料

### 目标
把仓库中的信息源分级，不再让交接文档、备份文档、正式基线混成一锅信息粥。

### 一级：主事实源
这些文件应作为正式依据：
- 六类工程化基线文档
- `governance/` 中稳定治理规范
- `contracts/` 中 schema
- `policies/` 中 machine-readable 规则

### 二级：阶段参考材料
这些文件可参考，但不能直接替代正式基线：
- `.claude/CURRENT_WORKSPACE_STATUS.md`
- `.claude/NEW_SESSION_MASTER_CONTEXT.md`
- `.claude/NEXT_PHASE_EXECUTION_PLAN.md`
- `.claude/OFFICIAL_COMPATIBILITY_AND_PRACTICE.md`

### 三级：历史档案
这些文件不参与当前事实判断：
- `*.bak.md`

### 完成标准
- 明确写出哪类文件是主事实源
- `.claude/` 被定义为交接与阶段材料
- `*.bak.md` 不再进入当前主线判断
- 后续讨论默认引用一级文档

### 风险
- 每次新会话又被旧 handoff 文档带偏
- 备份文件重新污染当前真相

## 7. P0-05：收敛 `README.md` 为统一入口

### 目标
让仓库一打开就能看懂，不需要口头补充一大堆背景。

### `README.md` 应至少回答
1. Xuanzhi-Core 是什么
2. 它不是什么
3. 当前技术栈有哪些核心组件
4. 仓库目录如何分层
5. 六类基线文档在哪里
6. 当前阶段大致处于哪一步

### 完成标准
- 新人打开仓库能获得正确第一印象
- README 不再沿用过时技术栈表述
- README 不再承担过多细节手册职责
- README 能把读者引导到正式基线文档

### 风险
- 仓库入口继续输出旧叙事
- 后续所有整理成果被 README 的旧地图拖后腿
