# 玄织治理中枢系统需求说明书
## Xuanzhi Governance Core System Requirements Specification
## Version: v0.1-draft

---

## 1. 文档目的

本文档用于定义“玄织”治理中枢系统第一阶段的正式需求边界、系统目标、角色分工、工作区结构、核心能力、非功能性要求、约束条件与阶段性交付要求。

本文档面向以下对象：

- 系统设计者
- 主控 agent 设计与治理维护者
- 开发执行器集成设计者
- 工作流/自动化接入设计者
- 后续 schema / policy / validator 实施者

本文档的作用不是直接给出实现细节，而是明确：

- 系统要解决什么问题
- 系统不解决什么问题
- 系统边界在哪里
- 第一版必须成立的最小能力是什么
- 后续实施应如何不跑偏

---

## 2. 系统背景与问题定义

### 2.1 背景

当前目标不是构建一个“无所不能的顶层多智能体帝国”，而是构建一套适合个人/小团队使用的、可本地部署、可持续演化的 agent 治理中枢。

系统需要同时面对几类任务：

- 长线开发任务
- 日常知识库问答
- 信息采集与分析
- workflow 驱动任务
- 媒体生成任务（如图片、视频）
- 结果评估、汇报与发布类任务

这些任务不必全部在同一个执行器中完成，但需要一个统一的控制与汇报中心。

### 2.2 核心问题

如果没有统一治理中枢，系统会逐步出现以下问题：

- 任务入口分散
- 记忆与状态割裂
- 多执行器输出风格不一致
- 缺乏统一汇报与可追踪性
- 风险与权限边界不清
- 顶层 prompt 膨胀
- 规则散落在多处且难以维护

因此需要一个治理中枢，统一负责：

- 入口
- 分发
- 摘要
- 汇报
- 记忆
- 风险判断
- 约束收敛

## 3. 系统定位

### 3.1 核心定位

玄织首先是：

- 治理中枢
- 控制平面
- 统一入口
- 汇报与记忆收敛中心

玄织其次才是：

- 秘书长
- 幕僚
- 学者型架构顾问

### 3.2 非定位

玄织不是：

- 女仆
- 恋爱对象
- 无条件执行器
- 情绪安抚机
- 顶层全能执行器
- 长线开发的直接主执行面

### 3.3 第一阶段架构定位

第一阶段采用：

- 主控 agent：OpenClaw / 玄织
- 开发执行器：Claude Code
- workflow/前台：FastGPT（可选承担部分流程前台）
- 检索与记忆后端：QMD
- 模型路由与成本控制：Higress
- 代码与项目系统：GitLab CE
- 持久化轻存储：SQLite
- 部署底座：Docker

不在第一阶段目标内的内容包括：

- OpenClaw 顶层复杂 agent teams
- 顶层复杂安全编排体系
- 顶层长线多代理开发 orchestration
- 全面自动化的“无人值守自治系统”

## 4. 设计目标

### 4.1 总体目标

系统需要在以下目标之间取得平衡：

- 效率
- 经济
- 风险
- 长期价值

### 4.2 第一版目标

第一版的主要目标是：

1. 建立一个稳定的治理中枢
2. 保持顶层工作区轻量、清晰、可控
3. 使长线开发任务可以稳定下沉到 Claude Code
4. 使知识问答与 workflow 类任务可以被统一汇报和调度
5. 使细则默认走检索与机器约束，而非无限上浮到 prompt
6. 为后续 schema / policy / validator / state table 的硬化留出清晰接口

### 4.3 成功标准

如果第一版成功，应满足：

- OpenClaw 根目录标准文件保持薄且职责明确
- 主控能够完成任务 intake、分类、分发、汇总与记忆更新
- Claude Code 能作为默认开发执行器接受结构化任务包
- 日常问答、工作流、媒体类执行可以被纳入统一汇报视图
- 新增细则不会直接挤爆顶层上下文
- 关键规则开始可下沉为结构化契约

## 5. 用户与使用场景

### 5.1 目标用户

目标用户为：

- 个人开发者
- 小团队主理人
- 需要多工具协作的知识工作者
- 同时处理开发、内容、自动化流程的独立操作者

### 5.2 核心使用场景

#### 场景 A：长线开发任务
用户提出软件开发任务，玄织完成：

- intake
- 任务分级
- 风险判断
- 选择 Claude Code 作为执行器
- 下发结构化任务包
- 接收阶段结果
- 汇总 review / 风险变化 / 下一步建议

#### 场景 B：知识库问答
用户提出项目、规则、历史决策、文档相关问题，玄织完成：

- 根层快速判断
- 必要时触发 QMD 检索
- 结合工作区记忆与检索结果回答
- 必要时更新摘要记忆

#### 场景 C：workflow 与自动化任务
用户提出信息采集、流程执行、内容生成、发布等任务，玄织完成：

- 任务分类
- 路由至 FastGPT 或专项执行系统
- 跟踪状态
- 汇总结果与工件引用
- 形成统一汇报

#### 场景 D：媒体生成任务
用户提出图片/视频生产需求，玄织完成：

- 确定任务类型与输出要求
- 路由至 ComfyUI 或视频 API 工作流
- 获取结果、评估摘要与工件引用
- 汇总并反馈

#### 场景 E：日常控制与汇报
用户希望知道当前系统在做什么，玄织完成：

- 汇总 running tasks
- 汇总当前 active executors / workflows
- 输出阶段摘要
- 提供日报/周报基础材料

## 6. 核心原则

### 6.1 薄主控原则

OpenClaw 顶层只做：

- 治理
- 分类
- 分发
- 摘要
- 记忆
- 汇报

不做：

- 长线开发细节推进
- 顶层复杂多代理执行编排
- 大量过程日志承载
- 重媒体流水线内部执行

### 6.2 薄根层原则

根目录标准文件必须：

- 短
- 高频
- 稳定
- 会话起始即值得加载

详细规则默认不进入根目录标准文件。

### 6.3 默认下沉原则

新增细则默认先放入：

- `governance/`
- `contracts/`
- `policies/`
- `integrations/`
- `memory/`

而不是根目录标准文件。

### 6.4 prose 与执行分离原则

Markdown 适合作为：

- 治理规范
- 说明性规则
- 语义基准

但不应长期独立承担：

- 严格字段校验
- 状态迁移 enforcement
- 风险门槛执行
- 准入判定
- 生命周期 gating

### 6.5 阶段回顾原则

每完成一个阶段，必须回顾：

- 是否增加了不必要复杂度
- 是否违反薄主控与薄根层原则
- 是否出现上浮失控
- 是否识别到需要硬化的规则

## 7. 总体架构需求

### 7.1 架构模式

系统应采用分层架构：

- 根目录标准文件层
- 治理细则检索层
- 机器契约层
- 策略/规则层
- 集成与执行层
- 记忆层

### 7.2 单一主工作区

系统应以 OpenClaw 单一活动工作区为控制中枢的根目录。

该工作区应作为：

- 标准文件承载层
- 工作区记忆层
- 检索入口层
- 控制层配置与长期摘要所在位置

### 7.3 工作区不是硬沙箱

系统设计必须承认：

- OpenClaw 工作区是默认 cwd
- 但不是天然硬隔离边界
- 真正隔离应由 sandbox 或执行器侧隔离实现

因此第一版需求中，不应把“工作区本身”误当成安全边界。

## 8. 工作区结构需求

### 8.1 根目录标准文件

系统必须支持并维护以下根目录标准文件：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `IDENTITY.md`
- `TOOLS.md`
- `HEARTBEAT.md`
- `MEMORY.md`
- `BOOT.md`
- `BOOTSTRAP.md`（初始化阶段临时存在）

### 8.2 根目录标准文件要求

这些文件必须：

- 保持短小
- 保持职责边界清晰
- 不承担长细则存储职责
- 不承担历史日志职责
- 不承担大规模项目知识归档职责

### 8.3 细则目录

系统必须支持以下细则目录结构：

- `governance/`
- `contracts/`
- `policies/`
- `integrations/`
- `workflows/`
- `memory/`

### 8.4 检索目录要求

`governance/` 与必要的 `integrations/` 文档应能够被纳入 QMD 或记忆搜索体系，以便按需检索，而不常驻注入上下文。

## 9. 主控 agent 需求

### 9.1 主控角色

主控 agent 必须承担以下职责：

- 统一任务入口
- 任务分类
- 复杂度与风险初判
- 执行器选择
- 任务包下发
- 结果包接收
- 汇总与汇报
- 长期记忆摘要维护

### 9.2 主控禁止职责

主控不应承担：

- 长线开发主执行职责
- 顶层复杂 agent teams 的直接组织
- 所有外部执行流程的细节处理
- 高频过程数据的直接长期存储

### 9.3 主控输出要求

主控输出必须尽量结构化，优先输出：

- 分类结果
- 风险摘要
- 执行器建议
- 任务包
- 结果摘要
- review 摘要
- 下一步建议

而非大量过程性 narration。

## 10. 开发执行器需求

### 10.1 默认开发执行器

Claude Code 应作为默认开发任务执行器。

### 10.2 开发执行器职责

Claude Code 侧必须负责：

- 接收结构化开发任务包
- 在 repo 中推进长线开发
- 进行阶段性代码实现与验证
- 在其内部按需要组织开发子代理或协作逻辑
- 输出阶段结果包

### 10.3 开发执行器边界

Claude Code 不承担：

- 顶层治理人格
- 顶层统一记忆维护
- 全局控制与汇报中枢职责

### 10.4 开发任务的结构化交互

系统必须设计以下机器可读契约：

- `dev_task_packet.schema.json`
- `dev_result_packet.schema.json`

用于主控与开发执行器之间的结构化交接。

---

## 11. Workflow 与外部执行需求

### 11.1 workflow 执行层

系统必须允许非开发任务由以下执行层承担：

- FastGPT
- ComfyUI
- 视频生成 API
- 发布流水线
- 其他外部自动化系统

### 11.2 workflow 接入原则

这些系统可以负责专项执行，但不得承担最终治理权。

### 11.3 主控与 workflow 的关系

主控必须能够：

- 调用或触发 workflow
- 追踪其状态
- 接收其摘要结果
- 引用其产物
- 统一汇报

---

## 12. 记忆与检索需求

### 12.1 记忆分层

系统必须采用至少两层记忆结构：

- `MEMORY.md`：长期记忆摘要
- `memory/YYYY-MM-DD.md`：每日运行上下文

### 12.2 长期记忆要求

`MEMORY.md` 必须仅保存：

- 稳定偏好
- 长期项目摘要
- 持久决策
- 少量导航信息

### 12.3 每日记忆要求

`memory/YYYY-MM-DD.md` 主要用于：

- 日常运行记录
- 临时上下文
- 阶段性备注
- 当日摘要

### 12.4 检索优先级

长规则、长说明、长项目材料应优先通过检索召回，而非写入根目录标准文件。

### 12.5 自动记忆刷新兼容性

系统设计应兼容 OpenClaw 的自动记忆刷新机制，但不得依赖其代替清晰的记忆写入政策。

---

## 13. 治理细则需求

### 13.1 治理细则层必须包含的主题

第一阶段需要至少建立下列治理主题：

- 根目录文件写入边界
- 规则硬化政策
- 术语总表
- 字段命名总表
- 状态机
- 风险模型
- trace 规范
- 开发任务模型
- 记忆写入政策

### 13.2 细则默认下沉

治理细则应默认写入 `governance/` 而不是根目录标准文件。

### 13.3 根目录文件写入政策

系统必须明确规定哪些内容不能进入根目录标准文件，以防未来迭代时失控上浮。

### 13.4 规则硬化政策

系统必须明确要求：

新增或修改细则时，必须评估其是否需要同步下沉为：

- JSON Schema
- YAML policy
- validator
- state transition table

## 14. 机器可读约束需求

### 14.1 契约层

系统第一阶段必须优先建立以下契约：

- `main-agent-step.schema.json`
- `dev_task_packet.schema.json`
- `dev_result_packet.schema.json`

### 14.2 策略层

系统第一阶段应规划以下策略文件：

- `state_transitions.yaml`
- `risk_policy.yaml`

### 14.3 后续扩展契约

后续应支持：

- `trace_event.schema.json`
- `agent_registry.schema.json`
- `workflow_registry.schema.json`

### 14.4 原则

凡是涉及以下内容的规则，必须优先评估机器化表达：

- 结构
- 必填项
- 枚举
- allow/deny 逻辑
- 审批门槛
- 风险门槛
- 状态迁移
- 注册表准入

## 15. 状态与风险需求

### 15.1 状态管理

系统必须支持至少以下概念层的状态管理：

- task
- step
- review
- approval

开发任务还应能扩展支持：

- epic
- milestone

### 15.2 风险管理

系统必须支持离散风险级别模型，并至少覆盖：

- 低风险
- 中风险
- 高风险
- 阻断级风险

### 15.3 阻断条件

系统应能识别以下类型的默认阻断因素：

- 明显 destructive
- 明显越权
- 明显 ROI 极低
- 明显信息不足且风险高

### 15.4 风险与执行器关系

高风险任务不能仅凭主控 prose 判断通过，应逐步转向策略层和审批层约束。

---

## 16. Trace 与可追踪性需求

### 16.1 基本要求

系统必须支持对关键任务过程进行结构化追踪。

### 16.2 追踪对象

至少应覆盖：

- task trace
- step trace
- review trace
- approval trace
- executor handoff trace
- memory write trace（后续）

### 16.3 不追踪内容

系统不得将原始完整 chain-of-thought 作为默认长期追踪对象。

### 16.4 追踪用途

追踪主要用于：

- 汇报
- 审计
- 排障
- 复盘
- 风险事件说明

---

## 17. 汇报与可观测性需求

### 17.1 统一汇报

系统必须能够统一汇总不同执行器和工作流的结果。

### 17.2 报表类型

第一阶段至少要为以下报表预留结构：

- 日报
- 周报
- 当前运行概览
- 风险事件摘要
- 执行结果摘要

### 17.3 运行状态视图

系统应能够呈现：

- 当前 running tasks
- 当前 active executors / workflows
- 最近失败任务
- 最近高风险动作
- 最近关键决策

---

## 18. 非功能性要求

### 18.1 可本地部署

系统必须优先支持本地部署。

### 18.2 中文可用性

系统必须支持中文工作流和中文用户交互。

### 18.3 开源优先

在可选情况下，优先考虑开源与自托管组件。

### 18.4 Token 节制

系统设计必须以控制上下文体积与 token 消耗为明确目标。

### 18.5 可演化性

系统必须支持后续迭代，而不依赖大规模推翻重做。

### 18.6 稳定性优先于炫技

在第一阶段，应优先选择：

- 稳定
- 可解释
- 可维护
- 可收敛

而不是：

- 炫目的顶层复杂编排
- 过早引入的抽象层
- 尚未被证明必要的 agent teams 设计

---

## 19. 约束条件

### 19.1 现有工具条件

当前已知工具与系统条件包括：

- GitLab CE
- QMD
- SQLite
- Claude Code
- Higress
- FastGPT
- Docker
- OpenClaw 内置工具体系

### 19.2 工作区兼容条件

系统需求必须兼容 OpenClaw 官方工作区标准文件机制。

### 19.3 第一版实施约束

第一版不得要求：

- 顶层复杂 agent teams
- 顶层复杂安全体系
- 完整无人自治
- 所有规则第一天就全部机器化

### 19.4 设计约束

任何新增文件、规则、流程、状态或执行器，必须能够回答：

- 为什么现在需要它
- 为什么不能更简单
- 为什么不能放在更低一层
- 是否会导致根层变胖
- 是否需要同时补机器约束

## 20. 阶段性交付要求

### 阶段 0：总护栏
交付：

- `ROOT_FILE_POLICY.md`
- `RULE_HARDENING_POLICY.md`
- 《OpenClaw 官方兼容工作区 v0.2》
- 《设计核心原则及注意事项》

目标：

- 建立反漂移护栏
- 明确分层蓝图
- 明确阶段回顾机制

### 阶段 1：根目录标准文件
交付：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `IDENTITY.md`
- `TOOLS.md`
- `HEARTBEAT.md`
- `MEMORY.md`
- `BOOT.md`

目标：

- 建立薄根层
- 保持 OpenClaw 顶层轻量与稳定

### 阶段 2：语义基准
交付：

- `GOVERNANCE_GLOSSARY.md`
- `FIELD_CANON.md`

目标：

- 统一术语
- 统一字段

## 21. 阶段回顾要求

每个阶段完成后，必须回顾：

1. 是否违背了薄主控原则
2. 是否违背了薄根层原则
3. 是否产生了不必要复杂度
4. 是否有内容上浮失控
5. 是否出现重复定义
6. 是否遗漏应硬化的规则
7. 是否仍符合当前“主控 + 开发执行器”的目标架构
8. 是否仍有利于中文 UI、本地部署与开源优先策略

如果发现明显跑偏，默认动作应是：

- 纠偏
- 下沉
- 压缩
- 补契约
- 再进入下一阶段

## 22. 验收标准

当第一版需求阶段完成时，应满足以下验收条件：

- 系统边界清楚
- 分层清楚
- 根目录标准文件职责清楚
- 开发执行器与主控边界清楚
- 检索层与常驻层边界清楚
- 至少一部分规则已经有明确的硬化路线
- 阶段推进与阶段回顾机制已建立

## 23. 最终说明

本需求说明书服务于第一版的“正确方向建立”。

它不要求第一版一次性完成所有复杂能力。

它要求的是：

- 先把结构搭对
- 先把边界钉住
- 先把根层变薄
- 先把主控做稳
- 先把细则与机器约束分层
- 先让后续迭代变轻松

真正高质量的第一版，不是内容最多的第一版。

而是最不容易在第二、第三轮演化时崩掉的第一版。
