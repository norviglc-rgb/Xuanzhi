# MEMORY.md

## 用途

本文件存储玄织工作空间的精选长期记忆。

它是摘要与导航层。

其工作是保存应该塑造跨会话未来判断的持久上下文。

它不是：

- 完整的项目档案
- 任务追踪器
- 运行执行日志
- 日常笔记本
- 完整的知识库目录
- 大量复制材料的存放处

详细或短命的笔记属于 `memory/YYYY-MM-DD.md` 或其他下层文档。

---

## 记忆姿态

记忆应该是：

- 持久的
- 跨会话有用的
- 低噪音的
- 摘要导向的
- 有选择性的
- 易于复用的
- 易于审查的

记忆不应成为临时上下文的垃圾堆。

更小更干净的记忆通常比更大更噪杂的记忆更有用。

---

## 什么属于这里

本文件应只保存可能超越当前时刻再次重要的信息。

合适的内容包括：

- 稳定的用户合作偏好
- 稳定的系统形态决策
- 摘要级的活跃重要倡议
- 持久的架构决策
- 重要的治理约束
- 重要的执行器边界决策
- 值得复用的重要教训
- 指向更深细节的导航指针

---

## 什么不属于这里

以下内容通常不应存储在这里：

- 临时任务状态
- 冗长的日常进展
- 完整对话记录
- 长推理链
- 完整变更历史
- 复制的源文档
- 重复的会议式笔记
- 详细实现日志
- 短命提醒
- 大量小事实列表
- 操作噪音

如果一条信息可能很快过期，它可能不属于这里。

如果一条信息只对一个活跃线程有用，它可能属于日常笔记而非长期记忆。

---

## 记忆类别

### 1. 稳定偏好

本节用于反复改善未来交互的持久合作偏好。

示例：

- 偏好批判性与第一性原理推理
- 偏好薄根文件
- 偏好将细节向下推入检索层
- 偏好治理清晰而非装饰性复杂

只有稳定偏好应出现在这里。

### 2. 系统形态决策

本节用于关于系统如何组织的持久架构选择。

示例：

- 玄织是治理核心
- OpenClaw 是主控制面工作空间
- Claude Code 是默认开发执行器
- 长时运行开发保持在 Claude Code 而非 OpenClaw 顶层
- 工作流与媒体执行可委托给专业系统

这些值得记忆，因为它们影响许多后续决策。

### 3. 活跃战略焦点

本节用于摘要级的主要持续方向。

本节应保持紧凑。

它可包括：

- 当前主要设计方向
- 活跃的架构清理焦点
- 高层级的当前实现阶段

它不应变成运行积压或详细路线图。

### 4. 持久教训

本节用于通过反复摩擦、失败或设计审查学到且可能后续仍有用的教训。

示例：

- 根层提示膨胀是真实风险
- 纯文本治理导致软控制
- 检索应吸收细节压力
- 重型执行应向下委托

教训应表述得足够简洁和通用以复用。

### 5. 导航指针

本节用于指向更深来源。

示例：

- 详细治理规范存在于 `governance/`
- 机器可读契约存在于 `contracts/`
- 机器可读规则存在于 `policies/`
- 执行器与工作流细节存在于 `integrations/` 与 `workflows/`
- 日常笔记存在于 `memory/YYYY-MM-DD.md`

指针是有用的。
大型嵌入细节则不是。

---

## 记忆写入规则

在写入本文件前，玄织应检查：

- 这是否跨会话持久
- 这是否可能再次重要
- 这是摘要级还是过程级
- 这是存在这里更好还是 `memory/YYYY-MM-DD.md`
- 添加这是否会使检索更清晰而非更噪杂

如果答案不确定，默认行动是不写入这里。

默认行动是将信息保持在更低或更临时的层。

---

## 记忆编辑规则

编辑本文件时，偏好：

- 凝缩而非累积
- 替换而非重复
- 摘要而非细节
- 类别清晰而非时间蔓延

本文件应随时间变得更易读，而非仅仅变长。

如果一节变得太大，应按含义拆分或下放到另一文件，仅保留简短指针于此。

---

## 当前稳定偏好

- 用户偏好批判性、第一性原理、反漂移的推理。
- 用户偏好具有干净结构骨架的强健首版。
- 用户偏好根层文件保持精简。
- 用户偏好详细规则向下移动到检索层而非膨胀提示上下文。
- 用户偏好重要规则在有理由时硬化为模式、策略、验证器逻辑或状态表。
- 用户偏好有纪律的合作而非服从表演。

---

## 当前系统形态决策

- 玄织是治理核心与顶层控制面。
- OpenClaw 是用于接收、分发、摘要、记忆纪律与报告的主工作空间。
- Claude Code 是长时运行开发工作的默认执行器。
- FastGPT 及其他工作流系统可用于适合的工作流面向任务，但它们不替代顶层治理。
- 专业媒体或自动化系统可执行窄域执行并向上报告。
- 工作空间应先向下生长，再向外生长。
- 根文件应保持精简、稳定且会话关键。

---

## 当前活跃战略焦点

- 构建一个与 OpenClaw 兼容的薄根层首版工作空间。
- 将长治理细节移动到可搜索的下层。
- 在扩展文件集之前建立强健的反漂移策略。
- 在玄织与 Claude Code 之间为长时运行开发创建稳定接口。
- 逐步将关键治理规则从纯文本形式转换为机器可读约束。

---

## 当前持久教训

- 提示安全不是控制；重要规则最终需要硬化。
- 没有契约与策略的详细 Markdown 治理层会漂移。
- 当根文件变成存储面而非控制面时会退化。
- 当重型执行被推到正确的执行器时，委托改善稳定性。
- 检索是核心设计工具，不是缺失纪律的后备方案。
- 阶段审查是必要的，因为漂移常以"合理添加"形式出现。

## 最终原则

本文件应帮助未来会话快速恢复系统形态。

如果它变得噪杂、过大或过于详细，它就失败了。

长期记忆应锐化判断，而非模糊它。
