# 首批发布缺口清单

说明：以下缺口基于 `Xuanzhi-Dev/spec/plans/release-requirements-trace-matrix.md`、`Xuanzhi-Dev/spec/requirements/open_claw多agent系统v1需求规格.md` 以及当前仓库事实整理，按严重级别从高到低排序。

| id | 关联需求 | 缺口描述 | 影响范围 | 严重级别 | 建议修复动作 | 验证标准 |
| --- | --- | --- | --- | --- | --- | --- |
| RG-01 | R-03 单一真相源；需求规格第 4 章、第 11 章、第 13 章 | 单一真相源的“正式落点”仍未进入当前 active runtime。仓库根目录当前只保留 `openclaw.json`、`agents/`、`hooks/`、`logs/`、`skills/`、`workspace-*`，而 `docs/system/`、`policies/`、`schemas/`、`state/`、`workflows/` 仍主要停留在 `Xuanzhi-Dev/legacy-root/` 或生成目录，导致系统级事实、治理规则、机器可执行规则与运行状态没有统一的正式承载面。 | 全系统：orchestrator、critic、architect、ops、skills-smith、agent-smith、claude-code、daily-* 全部受影响。 | P0 | 把 `docs/system/ARCHITECTURE.md`、`docs/system/GOVERNANCE.md`、`policies/*.json`、`schemas/*.json`、`workflows/*.json`、`state/*.json` 提升为当前 runtime 的唯一正式落点，并同步更新所有路径引用，避免继续依赖 legacy-root 作为事实来源。 | 根目录下的正式路径已存在且可被运行时直接读取；任一角色的 BOOT/AGENTS 只能引用这些正式路径；不存在必须回退到 `Xuanzhi-Dev/legacy-root/` 才能完成启动或路由的情况。 |
| RG-02 | R-04 工作流；R-11 用户实例创建；需求规格第 7 章、第 8 章、第 13 章 | 关键工作流仍缺少可执行证据链。虽然仓库里已经有 `materialize-core-agents`、`create-daily-user`、`memory-promote` 的样例/草案，但当前 active runtime 里没有同等明确的执行入口、调用约定和完整回放记录来证明这些 workflow 可以在正式环境中稳定执行并闭环写回 state/audit。 | ops、agent-smith、architect、critic、orchestrator，尤其影响用户开通和 core agent 物化链路。 | P0 | 将关键 workflow 物化到当前 runtime 的可执行入口，补齐触发方式、输入校验、状态流转、审计写入和 review 回流；为每条主流程保留一份可重放的端到端执行记录。 | 能从正式入口跑通 `materialize-core-agents`、`create-daily-user`、`memory-promote`，并在同一条链路里看到 state 更新、audit 记录和 critic review 结果。 |
| RG-03 | R-05 审计；R-11 用户实例创建；需求规格第 4 章、第 10 章、第 13 章 | 审计一致性仍未完全闭合。当前既存在 `logs/audit/*.jsonl` 的运行时样例，也存在 `state/audit/*.jsonl`、`~/.openclaw/logs/ops-guard.jsonl` 这类分散落点；不同流的字段来源、存放位置和回收方式并不统一，导致审计回看时需要跨目录拼接，缺少一个可稳定复用的统一审计 envelope。 | 审计检索、review-gate、ops 动作、用户实例创建、core agent 物化、memory promotion。 | P1 | 统一审计字段口径和主存储路径，明确 `requestId/source/target/action/decision/timestamp` 为最小公共字段，并规定哪些流写 `logs/audit`、哪些流写 `state/audit`、哪些流仅作为 hook 留痕。 | 任一审计事件都能用同一套字段在同一条查询链里重建；`requestId` 能串起动作、审批、状态更新和回流；不会再出现“有日志但不可关联”的情况。 |
| RG-04 | R-07 daily 用户隔离；R-11 用户实例创建；需求规格第 7 章、第 13 章 | daily 隔离运行证据不足。当前仅能看到单个示例用户 `test-user-001` 的 provisioning proof 和 Docker 样例，`openclaw.json` 的 `bindings` 仍为空，尚未证明两个及以上 daily 实例可以同时存在且不会共享长期记忆、workspace 或绑定状态。 | daily-<userId>、用户路由、记忆治理、绑定配置。 | P1 | 增补至少两个独立 daily 用户实例的运行样例，明确各自 workspace、agentDir、memory 路径和 bindings；补一条跨用户访问应被阻断的验证用例。 | 既能看到两个 daily 实例并存，也能验证它们的长期记忆、profile、state 和 binding 不互相串扰；跨用户读取/写入会被显式拒绝或隔离。 |
| RG-05 | R-09 skills / agent 管理；R-01 拓扑；需求规格第 5 章、第 11 章、第 13 章 | skills / agent 管理证据仍偏“静态配置”，缺少生命周期闭环。当前 `skills/xuanzhi-control` 只有控制文件和治理规则，`agents/` 只有 runtime 目录占位；但缺少一套可以证明“创建、更新、审查、登记、回滚”的真实工件链，也缺少对应的变更审计。 | skills-smith、agent-smith、相关 workflow/state/audit。 | P1 | 补齐 skill 与 agent 的创建/更新工作流，建立 catalog 与审计写入约定，并为至少一次 skill/agent 变更提供可回放证据。 | 能从输入到输出完整看到 skill/agent 的创建或变更、catalog 更新、审查记录和审计记录，且可按变更 ID 追溯。 |
| RG-06 | R-10 运维与部署；R-11 用户实例创建；需求规格第 1 章、第 9 章、第 11 章、第 13 章 | 部署 / 运维证据不足。`ops` 角色职责已写明，hooks 与 cron 也已存在，但仓库里还没有一条可直接执行的 deploy/install/health-check/lifecycle runbook 证据链，无法证明当前 runtime 已具备“可复制运维”的最低闭环。 | ops、release 流程、环境运维、生命周期管理。 | P1 | 增加最小运维 runbook 或 workflow，覆盖一次允许的安装/部署/巡检/恢复动作，并把结果写入审计和报告。 | 至少有一条受 allowlist 约束的运维动作可从启动、执行、记录到复核全链路复现，且结果能落到固定的 runbook / report / audit 工件。 |
| RG-08 | R-05 审计；R-10 运维 | `command-logger` 与 `boot-md` 在 Docker 动态验证里能正常注册，说明其为 OpenClaw internal hooks；但本仓库没有可审计的本地版本锚点与行为约束文档，后续升级时难以做差异审查和回归归因。 | 所有 agent 启动、命令审计、boot 文档流。 | P1 | 为 internal hooks 增加“版本锁定 + 行为基线”工件（最小行为快照、日志字段契约、升级回归清单），并把其纳入发布验收。 | 升级前后可对照同一份 hook 行为基线，确认关键日志字段和触发时机未漂移；若漂移可在工件中被识别并回滚。 |
| RG-09 | R-07 daily 用户隔离；R-11 用户实例创建 | `hooks/ops-action-guard/allowlist.json` 允许 `workspace-daily-*` 与 `agents/daily-*` 的动作，但当前 runtime 根目录的 `workspace-*` 只包括七个系统 workspace，没有任何 `workspace-daily-*` 或 `agents/daily-*`，导致 allowlist scope 无实际目标，daily 用户 provisioning 线无法落地。 | ops、agent-smith、critic、daily 实例管理。 | P1 | 实际创建至少一个 `workspace-daily-<userId>` 与 `agents/daily-<userId>/agent` 目录，补齐 `AGENTS.md`/`IDENTITY.md`/`MEMORY.md` 等文件，并产出 allowlist log，证明 `manage_daily_user_instance` 有实际运行对象；若暂不创建，应把 allowlist scope 约束缩减为现有目录并同步说明。 | 根目录出现 `workspace-daily-*` 与 `agents/daily-*` 目录（含必要文档），allowlist handler 的日志/审计记录能吐出对应 scope；or 说明 scope 被缩减后仍可控制 daily 相关操作。 |
| RG-10 | R-07 Docker 复制即用；R-10 运维与部署 | 复制运行时内容到 `~/.openclaw` 的现有临时步骤在 host 上保留了原始的 host 所有权，容器内运行 `chown -R node:node ~/.openclaw` 会因为权限被拒绝，导致自动化复现需要额外手动调整权限才能让运行时由容器用户持有。 | ops 运行流程、Docker 验收 proof、release repeatability。 | P2 | 在复制流程中使用容器用户创建/解压归档（或提前把 host 文件归 `node` 所属 uid/gid），确保 `./openclaw` 内的文件在容器里默认归 `node`，避免每次都需要手动 `chown`。 | 新的复制策略在干净环境跑一次就能在容器中由 `node` 读写 `~/.openclaw`，`chown` 命令不再报权限错误，且 downstream 验证任务不用预设额外权限。 |

## 说明

- P0 先解决“正式事实落点”和“工作流可执行证据”这两个发布拦截项。
- P1 解决审计、daily 隔离、skills/agent 管理、运维部署这四类上线前必须补齐的证据项。
- P2 作为复制即用与交付体验的收尾项，在不影响主链路判断的前提下补强。
